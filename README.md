# HW5 MLOps — Reproducible ML Experiment

## Цель проекта

Построить минимальный MLOps-контур с воспроизводимым экспериментом: контроль версий данных (DVC), автоматизация пайплайна обучения, логирование экспериментов (MLflow).

## Структура проекта

```
├── data/
│   ├── raw/          # Сырые данные (версионируются через DVC)
│   └── processed/    # Подготовленные данные (генерируются пайплайном)
├── src/
│   ├── prepare.py    # Подготовка данных (train/test split)
│   └── train.py      # Обучение модели + логирование в MLflow
├── dvc.yaml          # Описание пайплайна
├── dvc.lock          # Фиксация версий пайплайна
├── params.yaml       # Гиперпараметры
├── requirements.txt  # Зависимости
└── README.md
```

## Как запустить

```bash
git clone <repo-url>
cd HW5
pip install -r requirements.txt
dvc pull
dvc repro
```

## Описание пайплайна

Пайплайн состоит из двух стадий (dvc.yaml):

1. **prepare** — читает `data/raw/data.csv`, делает train/test split согласно `params.yaml`, сохраняет в `data/processed/`
2. **train** — обучает RandomForestClassifier, логирует параметры и метрики в MLflow, сохраняет `model.pkl`

Параметры пайплайна (`params.yaml`):
- `split_ratio: 0.2`
- `random_state: 42`
- `n_estimators: 100`

## MLflow UI

```bash

```

В разделе **Experiments → iris_experiment** будут видны параметры, метрики (accuracy) и артефакт (model.pkl).

## Шаг 1. Ограничения Jupyter/Colab по сравнению с Marimo

### Ограничения традиционных ноутбуков (Jupyter/Colab)

1. **Скрытое состояние и порядок выполнения.** Ячейки можно запускать в произвольном порядке. Переменная, созданная в ячейке 5, может быть перезаписана ячейкой 2, если запустить их не по порядку. Это главная причина нарушения воспроизводимости.

2. **Не подходят для DevOps/CI-CD.** `.ipynb` — это JSON-файл с метаданными, выводами и состоянием ядра. При сравнении через `git diff` он нечитаем. Автоматически запускать его в пайплайне сложнее, чем `.py`-файл.

3. **Глобальное состояние ядра.** Если перезапустить только часть ячеек, ядро хранит старые значения переменных. Это приводит к ошибкам, которые воспроизвести сложно.

4. **Сложность тестирования.** Код в ячейках не является модулем — его нельзя импортировать и протестировать через pytest без дополнительных инструментов.

### Преимущества Marimo

- Реактивная модель: изменение одной ячейки автоматически пересчитывает зависимые.
- Файл `.py` — обычный Python-скрипт, читаемый в git diff.
- Детерминированный порядок выполнения — нет скрытого состояния.
- Легко интегрируется в пайплайны как обычный скрипт.

### Вывод

Для этапа DevOps предпочтительнее `.py`-файлы (Marimo или обычные скрипты): они версионируются чисто, запускаются в CI/CD без изменений и не имеют проблемы скрытого состояния. Jupyter/Colab удобен для исследовательского анализа, но не для воспроизводимого пайплайна.

## Шаг 4. Готовность ML-системы к продуктивному развёртыванию

### Из каких элементов может состоять ML-система

Полноценная ML-система включает:

1. **Data pipeline** — сбор, валидация, очистка и версионирование данных
2. **Feature Store** — централизованное хранилище признаков (Feast, Tecton)
3. **Training pipeline** — автоматизированное обучение с логированием (MLflow, W&B)
4. **Model Registry** — версионирование и управление моделями
5. **Serving layer** — API для инференса (FastAPI, TorchServe, Triton)
6. **Monitoring** — отслеживание дрейфа данных и деградации модели
7. **CI/CD pipeline** — автоматическое обучение и деплой при изменениях
8. **Контейнеризация** — Docker/Kubernetes для воспроизводимой среды

### Нужны ли все элементы на данном проекте

Нет. Для учебного проекта на датасете Iris достаточно пунктов 1–4. Serving, мониторинг и Kubernetes избыточны — датасет статичный, нагрузки нет, дрейфа данных не ожидается.

### Готовность к деплою: только репозиторий + Colab

**Не готова.** Причины:

| Элемент | Статус при Colab+репозиторий |
|---|---|
| Воспроизводимость среды | Нет — версии библиотек не зафиксированы |
| Версионирование данных | Нет — CSV в Git или нигде |
| Автоматизация обучения | Нет — запуск вручную в ноутбуке |
| Логирование экспериментов | Нет — результаты только в выводе ячеек |
| Serving / API | Нет |

**Что даёт данный проект (DVC + MLflow):**
- Зафиксированы версии данных и пайплайна через DVC
- Параметры и метрики логируются в MLflow
- Среда воспроизводима через `requirements.txt` + `dvc repro`
- Проект клонируется и запускается тремя командами

Для перехода в продакшн дополнительно нужны: serving API, мониторинг, CI/CD и контейнеризация.
